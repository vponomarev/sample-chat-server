# Практики отказоустойчивости

Документ разбит на две части:

- **Часть A — что уже есть в проекте** (и в каком оно состоянии: работает / заявлено, но
  сломано).
- **Часть B — что стоит добавить** (отдельный список, как просили) — сгруппировано по
  уровням системы.

Для учебного курса полезно проговаривать не только «какой паттерн», но и **от какого
типа отказа он защищает**.

## Часть B. Практики, которые стоит добавить

Список того, чего в проекте нет (или что стоит довести до рабочего состояния как новую
практику). Отсортировано по уровням; в скобках — приоритет для учебного трека.

### B1. Доставка и согласованность данных

1. **Межсерверная доставка сообщений (fan-out / pub-sub)** — чтобы кластер реально был
   кластером, а не тремя изолированными чатами. Простой вариант: мастер форвардит
   сообщение всем узлам, каждый отдаёт своим клиентам. (высокий)
2. **Серверная идемпотентность по `client_msg_id`** — уникальный индекс + возврат уже
   присвоенного `msg_id` вместо повторной вставки. Даёт «эффективно ровно один раз».
   (высокий)
3. **Гарантии доставки WAL: ACK и повторная отправка** — сейчас репликация
   «выстрелил и забыл». Добавить подтверждение применения и ретрай недоставленного.
   (средний)
4. **Кворум/подтверждение записи (по желанию)** — не отвечать клиенту `ACK`, пока запись
   не реплицирована на большинство узлов (синхронная репликация) — обсудить компромисс
   «скорость vs. надёжность». (продвинутый)

### B2. Кластер и лидер

5. **Кворум/большинство для становления мастером + fencing по `term`** — защита от
   split-brain (проблема #11). Ключевая тема курса. (продвинутый)
6. **Периодическая проверка живости мастера** (не только реакция на heartbeat-событие) —
   надёжнее запускает выборы. (средний)
7. **Аутентификация межсерверного трафика** (shared-token/mTLS) — против «захвата»
   кластера (#10). (высокий)
8. **Явная конфигурация топологии** (`server_id` + адрес каждого узла) вместо угадывания
   ID по позиции (#5). (высокий)
9. **Jitter в ретраях и heartbeat** — против «thundering herd»/шторма выборов (#21).
   (средний)

### B3. Устойчивость сервера

10. **Persistent-сессии / токены аутентификации (JWT или серверный session-store)** —
    чтобы переживать реконнект и рестарт сервера; клиент переавторизуется автоматически
    (#6). (высокий)
11. **Тайм-ауты, ретраи и circuit breaker для вызовов к пирам** — чтобы медленный сосед
    не блокировал узел. (средний)
12. **Rate limiting и валидация ввода** (макс. длина сообщения, частота) — против флуда
    (#17). (средний)
13. **Backpressure / лимит подключений / load shedding** — контролируемая деградация под
    нагрузкой. (средний)
14. **Явные транзакции и обработка ошибок БД** — атомарность многошаговых операций,
    грациозная деградация при сбое диска (#18). (средний)
15. **Graceful draining при остановке** — дать активным операциям завершиться, уведомить
    клиентов о переезде. (низкий)

### B4. Данные и восстановление

16. **Резервное копирование + снапшоты БД** и **компакция/усечение WAL** — чтобы журнал
    не рос бесконечно и было откуда восстановиться. (средний)
17. **Процедура восстановления узла с нуля** (bootstrap из снапшота + догон по WAL) —
    отработать как отдельный сценарий. (средний)
18. **Проверки целостности/контрольные суммы реплик** — обнаружение расхождений данных.
    (низкий)

### B5. Наблюдаемость и проверка отказоустойчивости

19. **Починить и расширить метрики** (см. #8, #9): подключения, комнаты, реальный lag;
    добавить RED-метрики (Rate/Errors/Duration) на команды. (высокий — дёшево и наглядно)
20. **Correlation ID / трассировка** сообщения «клиент → сервер → реплики». (низкий)
21. **Разделить liveness и readiness пробы** — узел «жив», но «не готов принимать трафик»
    (например, догоняет WAL). (средний)
22. **Chaos-тестирование / инъекция отказов** — скрипты, которые роняют мастер, режут
    сеть, добавляют задержки; интеграционные тесты на failover. Идеальное учебное
    задание. (высокий для курса)
23. **Нагрузочное тестирование** — найти предел и посмотреть на деградацию. (низкий)

### B6. Транспорт и безопасность

24. **TLS / `wss://`** для клиента и кластера. (средний)
25. **Ужесточить CORS и авторизацию** (#16), обязательные пароли (#15). (низкий)

## Учебная карта «отказ → защита»

Полезная сводка для урока: какой сбой каким паттерном закрывается.

| Тип отказа | Уже есть | Стоит добавить |
|------------|----------|----------------|
| Кратковременный обрыв сети клиента | reconnect + backoff, офлайн-очередь | jitter, авто-релогин (#6/B10) |
| Падение одного сервера | ротация серверов на клиенте | рабочие выборы (#3), fencing/кворум (B5) |
| Потеря/дубли сообщения | `client_msg_id`+ACK на клиенте | серверная идемпотентность (B2), ACK репликации (B3) |
| Потеря данных при отказе узла | WAL-режим SQLite | рабочая репликация (#1), кворум-запись (B4) |
| Перегрузка/флуд | — | rate limit, backpressure (B12/B13) |
| Сетевое разделение (split-brain) | — | кворум + fencing (B5) |
| Медленный сосед в кластере | — | таймауты + circuit breaker (B11) |
| Тихий отказ (данные разъехались) | — | сверка реплик, снапшоты, chaos-тесты (B18/B16/B22) |
