## 🔴 Critical

### #1. WAL-репликация не подключена к записи (реплики никогда не получают данные)
- **Где:** `server/irc/commands.py` (все `handle_*`), `server/cluster/replication.py:135`
- **Симптом:** таблица `wal` всегда пустая; slave-узлы не получают ни пользователей,
  ни комнат, ни сообщений.
- **Причина:** обработчики команд пишут в БД напрямую — `await self.db.execute("INSERT …")`
  (`commands.py:58, 172, 300, 453` и т.д.) и **ни разу** не вызывают
  `replication.log_operation()` / `WALWriter`. Поле `self.cluster` в `CommandHandler`
  сохраняется (`commands.py:20`), но не используется.
- **Проверить:** поднять кластер, отправить сообщение на server1, выполнить на server2
  `SELECT COUNT(*) FROM messages;` — будет 0. Также `SELECT COUNT(*) FROM wal;` на
  server1 = 0.
- **Чинить:** пропускать все мутирующие операции мастера через `log_operation()`
  (INSERT/DELETE в `users/rooms/room_members/messages`), которая пишет и в основную
  таблицу, и в `wal`, и рассылает на реплики.

### #2. Heartbeat опрашивает `/health` вместо `/cluster/health` → роль соседа неизвестна
- **Где:** `server/cluster/heartbeat.py:33` (`health_url = {url}/health`), `:128`;
  простой `/health` определён в `server/network/routes.py:66` и возвращает только
  `{"status": "ok"}`.
- **Симптом:** `peer.role` навсегда остаётся `"slave"`; `is_master_alive()` всегда
  `False`, `get_master()` всегда `None`.
- **Причина:** роль/терм/uptime отдаёт другой endpoint — `/cluster/health`
  (`server/cluster/peer_handler.py:32`), но heartbeat туда не ходит.
- **Следствие → см. #3.**
- **Чинить:** в `ServerInfo.health_url` использовать `/cluster/health`.

### #3. Выборы не запускаются при падении мастера
- **Где:** `server/cluster/manager.py:168` (`_on_peer_down`).
- **Симптом:** мастер упал — новый лидер не выбирается, кластер «замирает».
- **Причина:** `_on_peer_down` стартует выборы только если `peer.role == "master"`,
  но из-за #2 роль соседа всегда `"slave"` → условие ложно.
- **Чинить:** после исправления #2 условие заработает; дополнительно стоит запускать
  выборы и когда пропал узел с наибольшим ID, и по периодической проверке
  `is_master_alive()`.

### #4. Сообщения не расходятся между узлами кластера
- **Где:** `server/irc/commands.py:475` (`broadcast_to_room`), `server/network/ws_manager.py:72`.
- **Симптом:** пользователи, подключённые к разным серверам одной комнаты, не видят
  сообщений друг друга в реальном времени.
- **Причина:** рассылка идёт только по локальному словарю `room_connections`. Механизма
  «переслать сообщение соседям, чтобы те отдали своим клиентам» нет. WAL (даже будь он
  подключён по #1) применяет запись в БД slave, но **не пушит** её в WebSocket-клиентов.
- **Чинить:** ввести межсерверную доставку сообщений (форвардинг на пиров или
  publish/subscribe), и на принимающем узле вызывать `broadcast_to_room`.

### #5. Неверное присвоение `server_id` соседям → ломается Bully-алгоритм
- **Где:** `server/config.py:37` — `"server_id": f"server{len(peers) + 2}"`.
- **Симптом:** на узлах, кроме server1, соседи получают неправильные ID.
- **Причина:** парсер игнорирует реальные имена и нумерует соседей как
  `server2, server3, …` в порядке перечисления, предполагая, что «я — server1».
  Пример из `docker-compose.yml:34`: у server2 `PEERS=server1:8080,server3:8082`, и
  server1 будет ошибочно назван `"server2"`, а server3 — `"server3"`. Bully выбирает
  лидера по числовому ID (`election.py:78`), поэтому неверные ID = неверный лидер /
  расхождения.
- **Чинить:** передавать `server_id` соседа явно (например `server1=host:port,…`) или
  выводить ID из ответа `/cluster/health`, а не из позиции в строке.

### #6. Клиент не переавторизуется после переподключения
- **Где:** `client/js/connection.js:75` (`onopen`), `client/js/app.js:328`
  (`_handleServerMessage`).
- **Симптом:** после разрыва/failover статус «🟢 Подключено», но любое действие
  возвращает `Authentication required`; retry-очередь пытается слать `MSG` в
  неаутентифицированный сокет.
- **Причина:** сессии на сервере хранятся в памяти и привязаны к сокету
  (`ws_manager.py:17`). Новый сокет = новая неаутентифицированная сессия. В `onopen`
  клиент лишь разбирает очередь, но заново `LOGIN` не отправляет.
- **Чинить:** после `onopen` автоматически переотправлять `LOGIN` по сохранённой сессии
  (в идеале — по токену, а не по паролю); до успешного `OK` не флашить очередь.

## 🟠 Major

### #7. Дублирующийся и сломанный файл `websocket_handler.py`
- **Где:** `server/network/websocket_handler.py`.
- **Проблемы:** импортирует `WebSocketHandler` из самого себя
  (`from network.websocket_handler import WebSocketHandler` — такого класса там нет);
  на строке 52 обращается к необъявленной переменной `ws_manager` → `NameError` при
  вызове. Файл нигде не импортируется (роуты используют `ws_handler.py`).
- **Чинить:** удалить файл. Заодно переименовать класс-менеджер `WebSocketHandler`
  (`ws_manager.py`) во что-то вроде `ConnectionRegistry`, чтобы не путать «менеджер
  подключений» с «обработчиком запросов».

### #8. Метрики объявлены, но не обновляются → дашборд и алерты «пустые»
- **Где:** `server/observability/metrics.py`.
- **Симптом:** `irc_connected_clients`, `irc_websocket_connections_total`,
  `irc_rooms_total`, `irc_rooms_active`, `irc_room_members` всегда 0.
- **Причина:** функции `update_connected_clients`, `increment_websocket_connections`,
  `update_rooms_*` **нигде не вызываются** (обновляется только счётчик сообщений и
  длительность команд). Алерт `NoConnectedClients` (`monitoring/alerts.yml`) поэтому
  срабатывает всегда/никогда — бесполезен.
- **Чинить:** вызывать соответствующие апдейтеры в `add_connection/remove_connection`,
  `create_room/delete_room`, `join/leave`.

### #9. `replication.get_lag()` всегда возвращает 0
- **Где:** `server/cluster/replication.py:354`.
- **Симптом:** метрика `cluster_replication_lag` бессмысленна; алерт `HighReplicationLag`
  никогда не сработает.
- **Чинить:** считать lag как `max_wal_seq(master) − last_applied_seq(slave)`.

### #10. Открытые кластерные endpoint'ы — возможность захвата кластера
- **Где:** `server/cluster/peer_handler.py`, `server/cluster/election.py` — нет никакой
  аутентификации (подтверждено: в `server/cluster/` нет ни `token`, ни `secret`).
- **Симптом:** любой, кто имеет сетевой доступ, может отправить
  `POST /cluster/election/coordinator {master_id: …}` и назначить произвольного мастера,
  либо залить фальшивые WAL-записи через `/cluster/replication/wal`.
- **Чинить:** общий кластерный секрет/токен в заголовке, ограничение источников,
  желательно mTLS. Для учебного стенда достаточно shared-token.
- **✅ Исправлено (Этап 3.1):** введён общий секрет `CLUSTER_SECRET`, узлы передают его в
  заголовке `X-Cluster-Token`; middleware (`server/cluster/auth.py`) отклоняет запросы к
  `/cluster/election/*` и `/cluster/replication/*` без верного токена (401). Сравнение —
  constant-time. Observability-ручки (`/cluster/health`, `/cluster/state`) намеренно
  открыты. Секрет задаётся через env или `config/config.yaml`. Не заменяет TLS
  (шифрования канала нет — см. Этап 4.3), но закрывает публичный доступ к управлению.

### #11. Нет защиты от split-brain (двойного мастера)
- **Где:** `server/cluster/election.py`, `manager.py:99` (`is_master=True` стартово у всех).
- **Симптом:** при сетевом разделении каждая половина может выбрать своего мастера;
  после «заживления» два мастера с расходящимися данными, нет сверки/fencing.
- **Причина:** Bully без кворума/большинства и без fencing-токена.
- **Чинить:** требовать большинство живых узлов для становления мастером; вводить term
  как fencing-токен и отвергать записи от «старого» мастера. (Это большая тема — см.
  [04-roadmap.md](04-roadmap.md), продвинутый этап.)
- **✅ Исправлено (Этап 4.1):**
  - *Кворум.* Узел становится/остаётся master только при строгом большинстве:
    `(живые, включая себя) * 2 > размер_кластера`. Для кворума живыми считаются
    лишь пиры, подтверждённые heartbeat'ом (`ServerInfo.ever_seen`) — иначе
    одинокий узел на старте оптимистично счёл бы всех живыми и стал master в
    меньшинстве. Реализовано: `ClusterManager._has_quorum`, гейт в
    `BullyElection._become_master`, а также step-down master'а при потере кворума
    в цикле `_update_metrics_loop`. Меньшинство остаётся без master (недоступно на
    запись), второй master не появляется — split-brain по записи невозможен.
  - *Fencing по term.* Master штампует свой `term` в каждом WAL-пуше
    (`replication._send_pending_to_peer`), реплика через `should_accept_term`
    отвергает пуш со `term` ниже уже виденного (`handle_wal_replication` →
    `409 Conflict`). «Зомби»-master, переживший разделение, после заживления сети
    не перезапишет данные, принятые от нового master с большим term.
  - Кворум виден в `/cluster/state` (`quorum: {has_quorum, alive, cluster_size,
    needed}`). Покрыто `TestQuorum`, `TestFencing` и интеграционным тестом
    `test_minority_node_never_becomes_master`.
  - Не покрывает: синхронную репликацию по кворуму (4.2) и сверку реплик (4.4).
    Замечание: полноценная отказоустойчивость к split-brain требует нечётного
    числа узлов (3+); кластер из 2 узлов теряет master при падении любого.

### #12. Реплика может нарушить порядок/ссылочную целостность WAL
- **Где:** `server/cluster/replication.py:272` (`_apply_insert`) + `database.py:24`
  (`PRAGMA foreign_keys=ON`).
- **Симптом:** если запись `messages` придёт/применится раньше, чем `rooms`/`users`,
  `INSERT` упадёт по внешнему ключу; ошибка проглатывается (`apply_wal_entry` возвращает
  `False`) и запись теряется без ретрая.
- **Чинить:** применять WAL строго по возрастанию `seq`, при пропуске — запрашивать
  недостающее через `/cluster/replication/sync`; не терять запись при ошибке.

### #13. Нет идемпотентности сообщений на сервере (дубликаты при ретраях)
- **Где:** `server/irc/commands.py:449` — на каждый `MSG` генерируется новый `msg_id`.
- **Симптом:** если `ACK` потерялся и клиент повторил отправку того же `client_msg_id`,
  на сервере создаётся второй `msg_id` → дубль в БД и у всех клиентов.
- **Чинить:** хранить `client_msg_id` (уникальный индекс), при повторе возвращать уже
  присвоенный `msg_id` без повторной вставки/рассылки.

## 🟡 Minor / чистота / безопасность

### #14. `config/config.yaml` не используется
- `server/config.py` читает только переменные окружения; YAML-файл не загружается
  (подтверждено поиском — `yaml` нигде не импортируется). Либо реализовать загрузку,
  либо удалить файл, чтобы не вводить в заблуждение.

### #15. Пароль необязателен
- `commands.py:52` — регистрация без пароля разрешена; при пустом пароле в БД
  (`user["password"]` пуст) `LOGIN` проходит без проверки. Любой знающий ник войдёт под
  чужим аккаунтом. Для демо приемлемо, но стоит хотя бы предупредить/сделать
  обязательным.

### #16. CORS полностью открыт
- `server/network/routes.py:17` — `Access-Control-Allow-Origin: *` на всё. Для учебного
  стенда ок, но стоит проговорить риск.

### #17. Нет ограничений ввода (длина сообщения, rate limit)
- `handle_msg` не ограничивает длину `text` и частоту → тривиальный флуд/переполнение БД.

### #18. Ошибки записи в БД не обрабатываются
- В `commands.py` вокруг `execute/commit` нет `try/except`; сбой БД всплывёт как общий
  `ERROR` без грациозной деградации. Многошаговые операции (например `handle_delete_room`,
  `commands.py:241–244` — 3 `DELETE` + `commit`) не обёрнуты в явную транзакцию.

### #19. Мёртвый код в репликации
- `replication.py:79,324` — очередь `_wal_queue` и цикл `_apply_wal_loop` создаются, но
  в очередь никто ничего не кладёт (WAL применяется напрямую из HTTP-хендлера
  `peer_handler.handle_wal_replication`). Цикл-потребитель бесполезен.

### #20. Дублирование роутов выборов
- `election.py:116` регистрирует `/election/*`, а `peer_handler.py:21` — `/cluster/election/*`.
  Обработчики частично дублируют друг друга; используется первый набор, второй сбивает с
  толку и рассинхронизирован с README (в README документированы `/cluster/election/*`).

### #21. `handle_election_start` провоцирует «шторм выборов»
- `election.py:203` — на каждое входящее ELECTION узел безусловно запускает свои выборы
  и отвечает `ok=True`, без подавления по уже идущим выборам/терму. В кластере это может
  давать лавину сообщений.

## Найдено при верификации клиента (браузер)

Эти дефекты не ловились юнит-тестами и всплыли при ручном прогоне клиента. Оба исправлены.

### #22. Статика клиента не отдавалась сервером (UI не грузился) 🔴
- **Где:** `server/network/routes.py` (регистрация роутов), `client/index.html`.
- **Симптом:** через `http://localhost:8080` страница открывается, но все ресурсы
  (`/js/*.js`, `/css/style.css`, `/manifest.json`) отдают **404** — UI сломан.
- **Причина:** сервер монтировал статику только на `/static/`, а `index.html` ссылается
  на ресурсы относительно корня (`js/…`, `css/…`) → браузер запрашивает `/js/…`.
  Коварно: тесты `TestStaticFiles` проверяли **`/static/…`** (пути, которые браузер не
  использует), поэтому были зелёными при сломанном UI.
- **Исправлено:** статика отдаётся по реальным путям — `/js/`, `/css/`, `/icons/`,
  плюс `/manifest.json` и `/sw.js` из корня (Service Worker обязан быть в корне для
  `scope='/'`). Тесты `TestStaticFiles` переписаны на реальные URL + добавлен тест `/sw.js`.

### #23. Дубли сообщений на клиенте (и «зависшая» неотправленная копия) 🟠
- **Где:** `client/js/messages.js`, взаимодействие с IndexedDB (`storage.js`).
- **Симптом:** одно отправленное сообщение отображается 2–3 раза, одна из копий
  застревает в статусе «отправка» (pending).
- **Причины:** (1) нет дедупликации по серверному `msg_id` — эхо-broadcast, история при
  JOIN/переподключении и перезагрузка комнаты рисовали копии; (2) по ACK `updateMessage`
  менял `msg_id` (это keyPath) и делал `put`, создавая **вторую** запись в IndexedDB и
  оставляя старую (pending) «осиротевшей»; (3) свои сообщения из истории отрисовывались
  повторно (усиливалось авто-реджойном при reconnect из #6).
- **Исправлено:** дедуп по `msg_id` (`seenMsgIds`) в отрисовке; ACK теперь заменяет
  временную запись канонической (ключ = серверный `msg_id`) и помечает id «увиденным».
- **Замечание:** дубли, попавшие в IndexedDB до фикса, остаются до очистки хранилища
  (DevTools → Application → Clear site data). После очистки новые не появляются.
