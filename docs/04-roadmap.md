# План развития и исправления

## Позиционирование (важно для приоритетов)

Проект — **референс «пример проекта с хорошими практиками»** для преподавателей.
Исходники ученикам не выдаются; возможна демонстрация отдельных компонентов. Отсюда
приоритеты:

- **Главная цель — сделать проект корректным и читаемым эталоном**, а не максимально
  «настоящей» распределённой системой. Если механизм заявлен в коде — он должен реально
  работать (иначе это плохой референс). Поэтому Этапы 0–2 обязательны.
- **SQLite остаётся намеренно** (снижение сложности) — не меняем на PostgreSQL и т.п.
- **Многосерверность (кластер) — ценное и уместное дополнение**, доводим до рабочего
  состояния (Этап 2).
- **Продвинутые темы (Этапы 3–4) — «приятно иметь»**: полезны как демонстрационные
  фрагменты, но это не обязательная планка. Гнаться за полной «взрослостью» и
  масштабом не нужно.
- Материалы пишем в тоне «вот как выглядит хорошая практика», **без привязки к
  конкретному занятию курса**.

План выстроен так, чтобы каждый этап был законченным и проверяемым: сначала чиним то,
что даёт видимый эффект «до/после» малой кровью, затем достраиваем реально работающий
кластер, и только потом — опциональные продвинутые темы.

Обозначения: приоритет 🔴/🟠/🟡, оценка сложности для школьников ★ (низкая) … ★★★ (высокая).
Для продвинутых этапов дополнительно помечаем: **[обяз.]** — нужно для корректного
референса, **[опц.]** — демонстрационное «приятно иметь».

## Этап 2. Кластер: заставить работать по-настоящему ✅ ВЫПОЛНЕНО

Реализована модель **primary + standby** (клиенты на master; реплики принимают
репликацию и отклоняют запись, перенаправляя клиента на master). Проверено на локальном
кластере из 3 узлов: репликация master→реплики и failover (убил master → выбран новый,
данные целы). Тестов: 95 зелёных.

| # | Задача | Статус | Ссылка |
|---|--------|--------|--------|
| 2.1 | Heartbeat опрашивает `/cluster/health` и читает роль/терм | ✅ | #2 |
| 2.2 | Явная конфигурация топологии `id@host:port`, без угадывания ID | ✅ | #5 |
| 2.3 | Выборы при падении master + периодическая проверка + step-down при split-brain | ✅ | #3 |
| 2.4 | WAL подключён к записи: мутации master через `log_operation()` + write-guard на репликах | ✅ | #1 |
| 2.5 | Применение WAL строго по `seq`, разрыв → догон через `/cluster/replication/sync` | ✅ | #12 |
| 2.6 | Доставка между узлами | ✅ (архитектурно: primary+standby, fan-out не нужен) | #4 |
| 2.7 | Реальный `replication_lag` (last_master_seq − last_applied) | ✅ | #9 |
| 2.8 | Консолидация роутов выборов `/cluster/election/*`, подавление шторма | ✅ | #20, #21 |

Также по пути исправлено: несовпадение HTTP-путей репликации (`/cluster/replication/*`),
раздельные БД для локальных узлов в `start_cluster.sh`, master переподтверждает лидерство
вместо абдикации при «мечущемся» узле.

**Проверка этапа (демо-сценарий failover):**
1. Три узла подняты, метрики зелёные.
2. Пользователь A на server1 и B на server2 в `#general` — видят сообщения друг друга.
3. Останавливаем мастер → в логах видны выборы → новый мастер за несколько секунд.
4. Клиенты переезжают (`SERVER_LIST`), чат продолжает работать, история цела.

## Рекомендованный порядок для курса (кратко)

```
   ОБЯЗАТЕЛЬНО (корректный референс)          ОПЦИОНАЛЬНО (демо для сильных групп)
   ┌──────────────────────────────────┐       ┌─────────────────────────────────┐
   Этап 0  →  Этап 1  →  Этап 2         │  ...  │  Этап 3 (безопасность/устойч.)  │
  (гигиена) (одиночный) (рабочий кластер)│      │  Этап 4 (кворум, fencing, TLS)  │
        │                               │       └─────────────────────────────────┘
        └──── Этап 5 (тесты отказов, параллельно) ────┘
```

**Минимальная планка для «хорошего референса» — Этапы 0–2** (всё заявленное реально
работает) плюс базовые проверки из Этапа 5. Этапы 3–4 — по желанию.

**Главная методическая идея:** после каждого этапа проводить демонстрацию отказа
(«убить сервер», «выключить сеть») и показывать поведение системы *до* и *после*
исправления. Именно этот контраст и есть предмет «построение отказоустойчивых систем».
