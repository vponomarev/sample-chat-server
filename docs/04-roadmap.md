# План развития и исправления

## Позиционирование (важно для приоритетов)

Проект — **референс «пример проекта с хорошими практиками»** для преподавателей.
Исходники ученикам не выдаются; возможна демонстрация отдельных компонентов. Отсюда
приоритеты:

- **Главная цель — сделать проект корректным и читаемым эталоном**, а не максимально
  «настоящей» распределённой системой. Если механизм заявлен в коде — он должен реально
  работать (иначе это плохой референс). Поэтому Этапы 0–2 обязательны.
- **SQLite остаётся намеренно** (снижение сложности) — не меняем на PostgreSQL и т.п.
- **Многосерверность (кластер) — ценное и уместное дополнение**, доводим до рабочего
  состояния (Этап 2).
- **Продвинутые темы (Этапы 3–4) — «приятно иметь»**: полезны как демонстрационные
  фрагменты, но это не обязательная планка. Гнаться за полной «взрослостью» и
  масштабом не нужно.
- Материалы пишем в тоне «вот как выглядит хорошая практика», **без привязки к
  конкретному занятию курса**.

План выстроен так, чтобы каждый этап был законченным и проверяемым: сначала чиним то,
что даёт видимый эффект «до/после» малой кровью, затем достраиваем реально работающий
кластер, и только потом — опциональные продвинутые темы.

Обозначения: приоритет 🔴/🟠/🟡, оценка сложности для школьников ★ (низкая) … ★★★ (высокая).
Для продвинутых этапов дополнительно помечаем: **[обяз.]** — нужно для корректного
референса, **[опц.]** — демонстрационное «приятно иметь».

## Этап 2. Кластер: заставить работать по-настоящему ✅ ВЫПОЛНЕНО

Реализована модель **primary + standby** (клиенты на master; реплики принимают
репликацию и отклоняют запись, перенаправляя клиента на master). Проверено на локальном
кластере из 3 узлов: репликация master→реплики и failover (убил master → выбран новый,
данные целы). Тестов: 95 зелёных.

|---|--------|--------|--------|

| 2.2 | Явная конфигурация топологии `id@host:port`, без угадывания ID | ✅ | #5 |
| 2.3 | Выборы при падении master + периодическая проверка + step-down при split-brain | ✅ | #3 |
| 2.4 | WAL подключён к записи: мутации master через `log_operation()` + write-guard на репликах | ✅ | #1 |
| 2.5 | Применение WAL строго по `seq`, разрыв → догон через `/cluster/replication/sync` | ✅ | #12 |
| 2.6 | Доставка между узлами | ✅ (архитектурно: primary+standby, fan-out не нужен) | #4 |
| 2.7 | Реальный `replication_lag` (last_master_seq − last_applied) | ✅ | #9 |
| 2.8 | Консолидация роутов выборов `/cluster/election/*`, подавление шторма | ✅ | #20, #21 |

Также по пути исправлено: несовпадение HTTP-путей репликации (`/cluster/replication/*`),
раздельные БД для локальных узлов в `start_cluster.sh`, master переподтверждает лидерство
вместо абдикации при «мечущемся» узле.

**Проверка этапа (демо-сценарий failover):**
1. Три узла подняты, метрики зелёные.
2. Пользователь A на server1 и B на server2 в `#general` — видят сообщения друг друга.
3. Останавливаем мастер → в логах видны выборы → новый мастер за несколько секунд.
4. Клиенты переезжают (`SERVER_LIST`), чат продолжает работать, история цела.

## Этап 4. Продвинутые темы (факультатив, «по-взрослому»)

Чисто демонстрационные вещи для сильных групп. Для целей проекта-референса не требуются;
включать точечно, если хочется показать, «как это делают в проде». Масштаб/эффектность —
не самоцель.

| # | Задача | Обяз./Опц. | Сложность | Ссылка |
|---|--------|-----------|-----------|--------|
| 4.1 | ✅ Кворум/большинство для мастера + fencing по `term` (анти-split-brain) | опц. | ★★★ | #11 |
| 4.2 | ✅ Синхронная репликация «запись подтверждается после кворума» | опц. | ★★★ | B4 |
| 4.3 | TLS/`wss://` для клиента и кластера | опц. | ★★ | B24 |
| 4.4 | ✅ Сверка целостности реплик (контрольные суммы) | опц. | ★★★ | B18 |

> **4.1 выполнено:** master становятся только при кворуме (строгое большинство
> `живые*2 > N`, где живыми для кворума считаются лишь пиры, подтверждённые
> heartbeat'ом — `ever_seen`). Master, потерявший кворум (меньшинство при
> сетевом разделении), сам слагает полномочия. Fencing по `term`: master
> штампует term в WAL-пуше, реплика отвергает пуш со старым term (`409`) —
> «зомби»-master после заживления сети не перезапишет данные нового master.
> Кворум виден в `/cluster/state`. Проверено юнит-тестами (`TestQuorum`,
> `TestFencing`) и интеграционным тестом `test_minority_node_never_becomes_master`
> (одиночный узел из 3 никогда не становится master). Замечание: кворум требует
> нечётного числа узлов (3+); кластер из 2 узлов теряет master при падении любого.

> **4.2 выполнено:** режим репликации переключается `REPLICATION_MODE`
> (env / `config.yaml`): `async` (умолч.) — master отвечает сразу, реплики
> догоняют фоном; `sync` — `log_operation` не возвращается (а значит, клиент не
> получает ACK), пока запись не примет большинство узлов (`_await_quorum_ack`
> опирается на подтверждённые seq из ACK-репликации 3.3). По таймауту запись
> остаётся на master с предупреждением (для стенда выбрана доступность; строгий
> вариант отклонял бы запись). Режим виден в `/cluster/state`. Покрыто
> `TestSyncReplication`. Побочно исправлена гонка сходимости на старте (младший
> узел принимает живого master по heartbeat, не инфлируя term и не устраивая
> шторм выборов) — иначе кворум-задержка становления master ломала выборы.

> **4.4 выполнено:** каждый узел детерминированно хеширует таблицы данных
> (sha256 по строкам в порядке первичного ключа — `cluster/checksum.py`) и отдаёт
> суммы по `GET /cluster/replication/checksum`. Master сверяет реплики через
> `GET /cluster/replication/verify`, различая *отставание* (seq реплики меньше —
> законно догоняет) и *расхождение* (seq не меньше, а содержимое отличается —
> реальная порча). Ловит «молчаливый» дрейф данных, который метрика lag не видит.
> Покрыто `TestChecksum` и интеграционным тестом
> `test_replica_integrity_checksums_match` (реальный кластер, суммы совпадают).

> Осознанно **вне рамок** проекта: смена SQLite на серверную СУБД, балансировка нагрузки
> и шардирование (темы курса 15–16). Их достаточно упомянуть как «следующий шаг» —
> реализовывать в этом референсе не нужно.

## Этап 5. Проверка отказоустойчивости как отдельная дисциплина (сквозной)

Идёт параллельно этапам 2–4 — лучший способ доказать, что «починили».

| # | Задача | Приоритет | Сложность | Ссылка |
|---|--------|-----------|-----------|--------|
| 5.1 | ✅ Интеграционный тест failover (3 реальных процесса, убить мастер, проверить выбор нового + сохранность данных) — `server/tests/test_failover_integration.py` | 🟠 | ★★ | B22 |
| 5.2 | ✅ Chaos-скрипты: рандомно ронять узлы, добавлять сетевые задержки/потери | 🟠 | ★★ | B22 |
| 5.3 | ✅ Нагрузочный тест: найти предел, посмотреть деградацию | 🟡 | ★★ | B23 |
| 5.4 | ✅ Correlation ID и трассировка пути сообщения | 🟡 | ★★ | B20 |

> **5.3 выполнено:** `scripts/loadtest.py` — asyncio-генератор нагрузки: поднимает
> одиночный сервер и гонит стадии с растущим числом одновременных клиентов (каждый
> шлёт M сообщений с замером латентности ACK), печатая таблицу throughput и
> перцентилей (p50/p95/p99). По ней виден потолок пропускной способности и рост
> хвоста латентности — «предел» и «деградация». Соединения устанавливаются
> батчами (фаза подключения отделена от замера, чтобы «стадо» не искажало
> результат); rate limit сделан конфигурируемым (`RATE_LIMIT_MSGS`) и поднят в
> бенчмарке. Матчасть (перцентили/сводка) покрыта `TestPercentile`/`TestStats`.
> Пример: `python scripts/loadtest.py --stages 10,50,100,200 --messages 20`.

> **5.2 выполнено:** `scripts/chaos.sh` поднимает кластер из 3 узлов и в цикле
> случайно роняет по одному узлу (SIGKILL), проверяя инвариант отказоустойчивости
> — среди живых ровно один master и у него кворум — затем поднимает узел обратно
> и ждёт возврата. Сетевые сбои переносимо (без root) инъектируются на уровне
> приложения через aiohttp-TraceConfig на межсерверных сессиях
> (`server/cluster/faults.py`): `CHAOS_LATENCY_MS` (задержка) и `CHAOS_DROP_PCT`
> (потеря = подвешивание дольше таймаута), по умолчанию выключено. Пример:
> `CHAOS_LATENCY_MS=150 CHAOS_DROP_PCT=5 ./scripts/chaos.sh 10`. Логика инъекции
> покрыта `TestChaosConfig`/`TestChaosTraceConfigs`; сценарий прогонялся вживую
> (переживает крах master под задержкой сети).

> **5.4 выполнено:** каждой входящей команде присваивается correlation id
> (`observability/correlation.py`, на `contextvars`): берётся из поля
> `correlation_id` запроса клиента или генерируется. Через `CorrelationIdFilter`
> id попадает во все логи, порождённые при обработке команды (включая логи
> репликации — та же задача), а `ConnectionRegistry` подмешивает его во все
> исходящие сообщения (ACK/MESSAGE/ERROR). Так весь путь запроса виден по одному
> `grep`. Покрыто `TestCorrelationId` и WS-тестами (эхо id клиента / генерация).

## Рекомендованный порядок для курса (кратко)

```
   ОБЯЗАТЕЛЬНО (корректный референс)          ОПЦИОНАЛЬНО (демо для сильных групп)
   ┌──────────────────────────────────┐       ┌─────────────────────────────────┐
   Этап 0  →  Этап 1  →  Этап 2         │  ...  │  Этап 3 (безопасность/устойч.)  │
  (гигиена) (одиночный) (рабочий кластер)│      │  Этап 4 (кворум, fencing, TLS)  │
        │                               │       └─────────────────────────────────┘
        └──── Этап 5 (тесты отказов, параллельно) ────┘
```

**Минимальная планка для «хорошего референса» — Этапы 0–2** (всё заявленное реально
работает) плюс базовые проверки из Этапа 5. Этапы 3–4 — по желанию.

**Главная методическая идея:** после каждого этапа проводить демонстрацию отказа
(«убить сервер», «выключить сеть») и показывать поведение системы *до* и *после*
исправления. Именно этот контраст и есть предмет «построение отказоустойчивых систем».
