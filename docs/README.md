# Анализ проекта Sample Chat Server

Этот каталог содержит результаты анализа демо-проекта «отказоустойчивого» IRC-подобного
чата (сервер на Python/aiohttp + браузерный клиент). Проект написан слабой LLM (qwen)
и предназначен для учебного курса «Построение отказоустойчивых систем» для 10–11 классов.

Цель анализа — не просто перечислить баги, а показать школьникам **разницу между
"практика упомянута в коде" и "практика реально работает"**. В проекте много правильных
идей (heartbeat, выборы лидера, репликация, очередь сообщений, метрики), но значительная
их часть **не подключена** или **работает неверно**. Это и есть главный учебный сюжет.

## TL;DR — три главных вывода

1. **Кластер выглядит рабочим, но фактически не реплицирует данные и не переживает
   падение мастера.** Записи в БД идут в обход WAL (`log_operation()` не вызывается ни
   разу), heartbeat опрашивает `/health` вместо `/cluster/health` и поэтому никогда не
   узнаёт роль соседа → выборы при падении мастера не запускаются. См.
   [02-issues.md](02-issues.md) #1–#3.

2. **Сообщения не расходятся между узлами кластера.** Рассылка идёт только локальным
   WebSocket-подключениям; межсерверной доставки нет. Два пользователя на разных узлах
   не видят друг друга — даже когда «кластер» формально поднят.

3. **Клиент переподключается, но не переавторизуется.** После reconnect/failover сокет
   на сервере снова «неаутентифицирован», а клиент повторно не логинится → все команды
   отклоняются с `Authentication required`. Отказоустойчивость соединения есть, а
   отказоустойчивости сессии — нет.

Проект отлично подходит как учебный полигон: почти каждый паттерн здесь можно
«починить» на уроке и увидеть разницу до/после.
