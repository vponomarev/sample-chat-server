# Чек-лист ручной проверки клиента (браузер)

Проверяет клиентскую отказоустойчивость, которую нельзя подтвердить юнит-тестами:
офлайн-очередь, авто-релогин после обрыва, выбор master и failover в кластере.

Легенда: **Шаги** → **Ожидаемо** → *(что проверяем)*.

Инструменты: браузер + DevTools (`F12` / `Cmd+Opt+I`). Полезны вкладки **Console**
(логи клиента с эмодзи) и **Network → троттлинг Offline/Online**. Для сброса состояния
между прогонами: **Application → Storage → Clear site data** (чистит IndexedDB и сессию).

---

## 0. Подготовка

**Вариант A — одиночный сервер** (для разделов 1–2):
```bash
./venv/bin/python server/main.py      # http://localhost:8080
```
Открыть `http://localhost:8080`.

**Вариант B — кластер из 3 узлов** (для раздела 3):
```bash
./scripts/start_cluster.sh            # server1..3 на портах 8081/8082/8083
curl -s localhost:8081/cluster/state  # убедиться: один master (server3), два slave
```
Открыть `http://localhost:8081`. Затем **⚙️ Настройки сервера** → host `localhost`,
port `8081` → сохранить. Клиент запросит `/api/servers`, увидит роли и сам подключится
к master. *(Без этого шага клиент по умолчанию идёт на 8080 и кластер не увидит.)*

---

## 1. Базовый смоук (одиночный сервер)

1. Вкладка **Регистрация** → ник `alice`, пароль `p` → **Зарегистрироваться**
   → появляется экран чата, статус **🟢 Подключено**. *(WS + REGISTER)*
2. Отправить сообщение в `#general` → оно появляется в ленте, статус меняется
   pending → sent (✓). *(MSG + ACK)*
3. Создать комнату `#test`, переключиться в неё, отправить сообщение. *(CREATE_ROOM/JOIN)*
4. Открыть **вторую вкладку** `http://localhost:8080`, войти как `bob`, зайти в `#general`
   → видно сообщения alice; сообщения bob появляются у alice в реальном времени.
   *(broadcast, история при JOIN)*

## 2. Этап 1 — надёжность клиента (одиночный сервер)

### 2.1 Офлайн-очередь сообщений
1. Войти как `alice`, быть в `#general`.
2. DevTools → **Network → Offline**.
3. Отправить 2 сообщения.
   → Сообщения показываются как **pending**; вверху виден **индикатор очереди** (📦 N).
   *(retry-очередь в памяти + IndexedDB)*
4. Network → **Online**.
   → Сообщения до­сылаются, статус → sent, индикатор очереди исчезает. *(flush очереди)*
5. (Персистентность) На шаге 3 при офлайне **перезагрузить вкладку** (F5), затем Online
   → неотправленные сообщения не потеряны (подхватились из IndexedDB). *(storage.js)*

### 2.2 Авто-релогин после обрыва — ключевая проверка (#6)
1. Войти как `alice`, находиться в `#general`.
2. Оборвать соединение: DevTools → **Network → Offline** (или на пару секунд остановить
   сервер `Ctrl+C` и снова запустить).
   → Статус **🔴 Офлайн**.
3. Вернуть связь: **Online** (или сервер снова поднят).
   → Статус **🟢 Подключено**, **и без ручного входа** можно сразу отправлять сообщения.
   *(до фикса тут было бы `Authentication required`)*
4. В **Console** ожидаемо: `🔑 Переавторизация после переподключения: alice`, затем
   повторный `JOIN` текущей комнаты; **не** должно быть `Authentication required`.
   *(onopen → авто-LOGIN → flush очереди только после OK)*
5. Отправить сообщение после переподключения → доходит, ACK получен. *(флоу целиком)*

### 2.3 Идемпотентность (наблюдательно)
1. При офлайне отправить сообщение (уйдёт в очередь), затем Online.
   → В ленте и в чужой вкладке сообщение появляется **ровно один раз**, дубля нет.
   *(серверный дедуп по `client_msg_id`)*

## 3. Этап 2 — кластер: master, редирект, failover

Требуется **Вариант B** (кластер) и настройка клиента на порт живого узла (см. §0).

### 3.1 Клиент подключается к master
1. Открыть клиент (настроен на `localhost:8081`), войти как `alice`.
2. В **Console** проверить список серверов: у одного `role: "master"` (server3, порт 8083).
   → Клиент подключён именно к master (при необходимости было `↪️ Переключение на master`).
   *(connection.connect() предпочитает master)*
3. Отправить сообщение → доходит (пишем на master). *(write на master)*

### 3.2 Реплика отклоняет запись и перенаправляет
1. В настройках сервера временно указать порт **реплики** (напр. `8081`, если master — 8083),
   сохранить, дождаться переподключения.
2. Отправить сообщение.
   → В **Console** виден `ERROR: Read-only replica` и приход `SERVER_LIST`; клиент
   автоматически переключается на master и сообщение проходит. *(write-guard + ensureOnMaster)*

### 3.3 Failover — главный демо-сценарий (#3/#4)
1. Войти как `alice`, переписываться в `#general` (несколько сообщений).
2. Узнать PID master и убить **только его** (пример для master на 8083):
   ```bash
   lsof -ti tcp:8083 -sTCP:LISTEN | xargs kill
   ```
3. Наблюдать:
   → Статус клиента ненадолго **🔴/🟡**, затем **🟢** — клиент переехал на новый master.
   → В `curl localhost:8081/cluster/state` новый master (server2, порт 8082) в течение
     ~10–15 c.
4. Продолжить переписку → работает; **история сообщений цела** (реплицирована). *(failover)*
5. (Опц.) Поднять упавший узел снова (`./scripts/start_cluster.sh` перезапускает все) —
   он входит как slave и догоняет WAL.

---

## На что смотреть в Console (маркеры фикса)
- `🔑 Переавторизация после переподключения: <nick>` — сработал авто-релогин (§2.2).
- `↪️ Переключение на master: {...}` — клиент ушёл на master (§3.1/3.2).
- `📦 Сообщение добавлено в очередь retry` / `✅ ... отправлено` — офлайн-очередь (§2.1).
- **Не** должно быть `Authentication required` после переподключения при активной сессии.

## Известные ограничения (ожидаемое поведение, не баги)
- **Перезагрузка страницы (F5)** теряет пароль из памяти → авто-релогин невозможен, нужен
  ручной вход. Полноценные токены сессии — отдельный пункт (Этап 3, roadmap B10).
- Во время выборов master (несколько секунд) запись недоступна — клиент ретраит с backoff.
- Всё по HTTP/`ws://` (без TLS) — учебный стенд.
